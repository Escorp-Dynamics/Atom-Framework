# Atom Buffers Toolkit

Этот пакет предоставляет высокопроизводительные инструменты для повторного использования выделяемых объектов и временных буферов в приложениях на .NET. Он состоит из трёх основных компонентов:

1. `PooledAttribute` — атрибут для декларативной генерации вспомогательного кода, упрощающего реализацию `IPooled`.
2. `ObjectPool<T>` — универсальный пул объектов с thread-local кешами и оптимизированным глобальным стореджем.
3. `SpanPool<T>` — выделение временных `Span<T>` блоков с возвращением в пул без лишних копирований.

Каждый компонент можно использовать отдельно, но вместе они закрывают типовые сценарии управления памятью при высоконагруженных задачах.

---

## 1. `PooledAttribute`

### Что делает

`PooledAttribute` вешается на метод сброса (как правило, `void Reset()`), объявленный внутри `partial`-класса. При компиляции `PooledMethodSyntaxProvider` генерирует файл `<TypeName>.Pooled.g.cs`, в котором:

- расширяет ваш тип, добавляя реализацию интерфейса `IPooled`;
- генерирует статические методы `Rent/Return`, использующие `ObjectPool<T>.Shared`;
- обеспечивает вызов помеченного метода при возврате экземпляра в пул.

Вам остаётся только описать «очистку» в методе, украсить его атрибутом и объявить класс `partial` — остальной код появится автоматически.

### Пример использования

```csharp
using System.Text;
using Atom.Buffers;

public sealed partial class JsonBuffer
{
    public StringBuilder Builder { get; } = new(capacity: 1024);

    [Pooled]
    public void Reset()
    {
        Builder.Clear();
    }
}
```

После сборки генератор создаст `JsonBuffer.Pooled.g.cs` примерно следующего содержания:

```csharp
public sealed partial class JsonBuffer : IPooled
{
    public static T Rent<T>() where T : IPooled =>
        ObjectPool<T>.Shared.Rent();

    public static JsonBuffer Rent() =>
        Rent<JsonBuffer>();

    public static void Return<T>(T value) where T : IPooled =>
        ObjectPool<T>.Shared.Return(value, static x => x.Reset());
}
```

Метод `Reset`, который вы написали, автоматически становится реализацией `IPooled.Reset()`.

### Требования к методу с атрибутом

- Сигнатура — `void MethodName()` без параметров. Имя не фиксировано, но традиционно используется `Reset`.
- Метод должен быть `public` или `internal`, чтобы генерация могла к нему обратиться.
- Класс, содержащий метод, обязан быть `partial` и иметь доступный конструктор (по умолчанию — конструктор без параметров).
- Внутри метода вы очищаете состояние, возвращаете вложенные ресурсы в их пулы и приводите объект к «нулевому» виду.

**Важно:** Код сброса должен быть идемпотентным и безопасным — `ObjectPool<T>` вызывает его каждый раз перед возвратом в пул. Исключения из метода приведут к выпадению арендуемого объекта из пула.

### Интерфейс `IPooled`

```csharp
public interface IPooled
{
    void Reset();

    static abstract T Rent<T>() where T : IPooled;

    static abstract void Return<T>(T value) where T : IPooled;
}
```

`PooledAttribute` избавляет от необходимости реализовывать эти члены вручную — генератор подставляет базовую реализацию, работающую через `ObjectPool<T>.Shared` и вызывающую ваш метод сброса.

## 2. `ObjectPool<T>`

### Общий обзор

`ObjectPool<T>` — быстрый lock-free пул, оптимизированный под сценарии:

- многопоточная аренда и возврат большого количества мелких объектов;
- thread-local кеш на фиксированный «батч» (по умолчанию 64 элемента), сводящий к минимуму кросс-поточные конфликты;
- глобальный массив с выравниванием по степени двойки (удобно подсчитывать индекс, легко расширять);
- поддержка фабрики `Func<T>` и ресеттера `Action<T>`.

### Быстрый старт

```csharp
// пул для StringBuilder
var pool = ObjectPool<StringBuilder>.Create(() => new StringBuilder(capacity: 1024));

var sb = pool.Rent();
try
{
    sb.Append("Hello pool!");
    // ... работа со StringBuilder ...
}
finally
{
    // можно вернуть с ресеттером
    pool.Return(sb, static builder => builder.Clear());
}
```

### Ключевые возможности

| Возможность                       | Подробности                                                  |
|-----------------------------------|--------------------------------------------------------------|
| `Create(Func<T> factory, int capacity)` | Пользовательская фабрика + указание начального размера (округляется до степени двойки). |
| `Create(int capacity)`            | Использование конструктора без параметров/`default(T)`. |
| `Shared`                          | Один процессный singleton, удобный для статических пуулов.   |
| `Rent(Func<T>)`                   | Позволяет временно переопределить фабрику. |
| `Return(T, Action<T>)`            | Возврат с ресеттером, гарантирует очистку thread-local и глобального кеша. |
| `Dispose()`                       | Полностью очищает структуры, дальнейшая аренда бросает `ObjectDisposedException`. |

### Производительность

- Thread-local сегмент уменьшает ложное sharing и снимает большую часть contention.
- Значительная часть API украшена `[MethodImpl(MethodImplOptions.AggressiveInlining)]`, чтобы минимизировать вызовы.
- Глобальный массив, выделенный через `GC.AllocateArray` с `pinned: true`, ускоряет обращения и дружит с span-паттернами.

При пропайке новых сценариев проверяйте тесты `ObjectPoolTests` (они находятся в `Tests/Atom.Tests/Buffers/`). Там описаны кейсы на конкурентную аренду, асинхронное использование с `Task.Yield()`, а также спецификации для `StringBuilder` — хорошая база для регрессий.

### Рекомендации

1. **Инициализация больших объектов** — используйте именованную фабрику, чтобы контролировать выделение буферов/массивов.
2. **Освобождение ресурсов** — если объект содержит `IDisposable` поля, позаботьтесь об их сбросе внутри ресеттера или `OnReturn()` (если тип поддерживает `IPooled`).
3. **Не храните ссылку** на пул внутри арендованного объекта — это может привести к удержанию объекта в памяти дольше необходимого при долгоживущих пулов. Лучше прокинуть пул в качестве параметра при возврате.

## 3. `SpanPool<T>`

### Задача

`SpanPool<T>` предназначен для сценариев, где нужно быстро арендовать буфер фиксированного размера и использовать его как `Span<T>` или `Memory<T>` без выделения на куче. Это особенно полезно для парсеров, сериализаторов и других операций с массивами.

### Использование

```csharp
using var lease = SpanPool<byte>.Shared.Rent(length: 4096);
Span<byte> buffer = lease.Span;

// работа с буфером
Fill(buffer);

// возврат произойдёт автоматически в Dispose leaser'a
```

`SpanPool<T>.Shared` — глобальный singleton, аналогичный подходу `ObjectPool<T>.Shared`. Для отдельного пула можно создать свой экземпляр:

```csharp
var pool = new SpanPool<char>(blockSize: 1024, blocksPerBucket: 16);
using var lease = pool.Rent(512);
```

### Особенности реализации

- Использует «ведёра» (`buckets`) — группы блоков определённого размера (обычно `2^N`). Это позволяет выдавать памяти побольше при необходимости и быстро возвращать блоки соответствующего размера.
- За счёт управления арендами через `struct`-обёртку (например, `SpanLease<T>`) нет дополнительного мусора и лёгкие проверки выхода за пределы.
- `SpanPool<T>` хорошо работает с `stackalloc` для маленьких буферов: можно адаптивно выбирать между стековыми и пуловыми буферами.

### Совместное использование с `ObjectPool`

Комбинирование пулов полезно, когда объекты содержат `Span<T>` внутри. Например, при реализации обработчика HTTP-запросов можно арендовать вспомогательные ресурсы в пользовательском методе и сбрасывать их в `[Pooled]`-методе:

```csharp
public sealed partial class RequestContext
{
    public StringBuilder? Builder { get; private set; }
    public SpanLease<byte> BufferLease { get; private set; }

    public void Initialize()
    {
        Builder ??= ObjectPool<StringBuilder>.Shared.Rent();
        BufferLease = SpanPool<byte>.Shared.Rent(2048);
    }

    [Pooled]
    public void Reset()
    {
        if (Builder is not null)
        {
            ObjectPool<StringBuilder>.Shared.Return(Builder, static sb => sb.Clear());
            Builder = null;
        }

        BufferLease.Dispose();
        BufferLease = default;
    }
}

// Использование
var context = RequestContext.Rent();
try
{
    context.Initialize();
    // ... работа с Builder/BufferLease ...
}
finally
{
    RequestContext.Return(context);
}
```

Сгенерированный файл `RequestContext.Pooled.g.cs` добавит реализацию `IPooled` и статические помощники `Rent/Return`, поэтому повторное использование контейнера сводится к вызову пары методов.

## Практические советы

1. **Профилирование** — измерьте потребление памяти и GC до/после интеграции пула, чтобы удостовериться в эффектах.
2. **Тестирование «грязных» возвратов** — добавляйте юнит-тесты, проверяющие, что объект после `Return()` вновь в «чистом» состоянии (например, тесты на `StringBuilder` в `ObjectPoolTests`).
3. **Адаптация под нагрузку** — не бойтесь экспериментировать с ёмкостью и размером батча. Для малых объектов `BatchSize = 64` и `DefaultCapacity = 1 << 20` достаточно, но для особых сценариев можно изменить.
4. **Безопасность** — при работе с небезопасным кодом (например, привязанные массивы) следите за корректным освобождением. Пул предоставит только переиспользование памяти, но не снимет требования к безопасной очистке.
